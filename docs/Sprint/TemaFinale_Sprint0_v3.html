Introduction

Goal: Costruire un sistema software cargoservice che automatizza le operazioni di carico di container nella stiva di una nave, impiegando un robot a guida differenziale (cargorobot).

Requirements

The company asks us to build a service named cargoservice that:

R1   is able to receive a request to load a container sent by some customer via the pushbutton of the IOPort.

  • It sends the answer retrylater if the IOPort is currently occupied by a container, or if the system is Out of service.
  • It rejects the request when the hold is already full (i.e. slots 1–4 are all occupied).
  • Otherwise, it considers the system as engaged, detects a free slot, and returns the name of the reserved slot. While engaged, the system must blink a LED.

R2   when the request to load is accepted, the customer must move the container into the sensor area within a prefixed time (e.g. 30 secs); otherwise the system becomes disengaged.

R3   uses the cargorobot to move the container from the IOPort to slot5 (for barcode marking the container) and then to the reserved slot.

R4   shows on the display of the IOPort:

  • the current state of the hold;
  • the message "Service working" when everything is going well;
  • the message "Out of service" if the sonar measures D > DFREE for at least 3 secs (possible sonar failure).

Requirement Analysis

Sistema cargoservice

Diamo il nome cargoservice al sistema software da realizzare, per distinguerlo dalle entità che lo compongono. Si tratta di un sistema distribuito su più nodi computazionali, i cui componenti operano tramite scambio di messaggi.

Il sistema comprende le seguenti macro-entità:

  • il cargorobot (DDR), da progettare e realizzare
  • il sensore sonar, dispositivo fisico da integrare
  • la stiva (hold), con i suoi slot di posizionamento
  • il marker, dispositivo che etichetta con barcode il container in slot5 e segnala il completamento
  • il dispositivo IOPort, dotato di pushbutton e display
  • il LED di allarme/stato, attuatore fisico

cargorobot (DDR)

Il DDR è un robot a guida differenziale che può muoversi in avanti, indietro, ruotare su se stesso e fermarsi. È disponibile un simulatore (VirtualRobot26) che espone i comandi-base del robot in cril, presentandosi come un servizio che fornisce solo primitive di movimento.

Il cargorobot come entità con comportamento applicativo va però progettato e realizzato da noi: la responsabilità dell'orchestrazione delle operazioni (raggiungere l'IOPort, prelevare il container, raggiungere slot5, ecc.) appartiene al cargoservice, non al DDR. Esiste dunque un abstraction gap significativo tra i comandi primitivi del simulatore e le operazioni di alto livello richieste, che sarà analizzato nella Problem Analysis.

Domanda: Il servizio VirtualRobot26 è sufficiente per risolvere il problema (posizionare un container in uno slot e tornare in HOME) o richiede modifiche?
Risposta: il simulatore fornisce le primitive necessarie (avanza, ruota, fermati) e che la logica applicativa vada implementata sopra di esse.

IOPort

L'IOPort è un dispositivo fisico collocato all'ingresso della stiva. È composto da:

  • un pushbutton: premuto dal cliente per inviare una richiesta di carico;
  • un display: mostra la risposta alla richiesta e lo stato corrente della stiva;

Pushbutton e display sono in realtà l'interfaccia fisica di una web GUI: il pushbutton invia richieste al sistema, il display ne mostra gli esiti. Il cliente vi accede da un qualsiasi dispositivo collegato alla rete (es. smartphone), puntando il browser all'IP del nodo che espone la pagina. Tale pagina può essere servita dallo stesso nodo che gestisce l'applicazione oppure da un microservizio ad hoc separato; per preferenza del committente si adotta questa seconda soluzione, trattando l'IOPort come nodo a sé stante (si veda Struttura del sistema).

Sonar sensor

Il sonar è un sensore fisico collocato davanti all'IOPort. Misura continuamente una distanza D. Il valore DFREE è una costante di configurazione che rappresenta la distanza libera attesa (area vuota davanti al sensore).

Due soglie operative:

  • D < DFREE/2 per almeno 3 secondi → presenza di un container allo IOPort → stato Service working
  • D > DFREE per almeno 3 secondi → possibile guasto del sensore → stato Out of service
Domanda: Il software che gestisce il sonar è già disponibile, oppure dobbiamo costruirlo noi?
Risposta: in attesa di uno strumento più evoluto, viene fornito un dispositivo fisico basato su un RaspBerry PicoW, senza software di supporto. Occorre scrivere il firmware sul microcontrollore.
Domanda: in quale formato ci aspettiamo di ricevere i dati dal sonar?
Risposta: stringhe della forma distance(DIRECTION, D), dove D è la distanza misurata (intero, in cm o mm).

Occorre valutare se sia sufficiente scrivere software sul microcontrollore per integrare compiutamente il sonar nel sistema, oppure se serva un componente intermediario (proxy/adapter) che traduca i messaggi grezzi in eventi comprensibili al cargoservice.

Domanda: Cosa accade se il guasto sonar si verifica mentre il cargorobot è già in movimento? Due interpretazioni: (A) il robot si ferma immediatamente e riprende dal punto in cui era al ripristino; (B) il robot completa l'operazione corrente e poi il sistema si mette in attesa.
Risposta: interpretazione A. Il robot si ferma immediatamente (halt), il container rimane nella posizione corrente e il sistema resta sospeso finché il sonar non torna a misurare D ≤ DFREE. Al ripristino, il robot riprende l'operazione dal punto in cui si era fermato.

Marker (slot5)

slot5 è una stazione di marcatura temporanea. Quando il cargorobot deposita un container in slot5, il dispositivo marker applica un barcode identificativo: attende alcuni secondi (tempo di marcatura) e poi segnala al sistema il completamento dell'attività. Solo allora il robot può prelevare il container e portarlo nello slot riservato.

Domanda: Come segnala il marker il completamento della marcatura? Via messaggio su rete?
Risposta: via messaggio, ma non necessariamente su rete. Infatti il marker è logicamente parte del nodo cargoservice (si veda Struttura del sistema), il messaggio può quindi restare interno al nodo.

Hold (stiva)

La stiva è un'area rettangolare e piatta con un unico punto di ingresso/uscita (IOPort). Fornisce:

  • slot1–slot4: slot per la deposizione definitiva dei container
  • slot5: stazione temporanea di marcatura

Formalizziamo la stiva come classe Java Hold:

public class Hold {
    public static final int NUM_SLOTS = 4;       // slot1..slot4

    public enum SlotState { FREE, RESERVED, OCCUPIED }
    // RESERVED: slot assegnato a una richiesta accettata, robot non ancora arrivato.
    // Finché uno slot è RESERVED il sistema è 'engaged' e non accetta nuove richieste.

    private SlotState[] slots = { // slot1..slot4
        SlotState.FREE,
        SlotState.FREE, 
        SlotState.FREE,
        SlotState.FREE
    };

    // slot5 è libero o occupato dal marker
    private boolean slot5Occupied = false;

    // Riserva il primo slot libero; restituisce il suo nome (es. "slot1") o null se piena
    public String reserveFirstFree() { ... }

    // Segna lo slot attualmente RESERVED come OCCUPIED
    public void markReservedOccupied() { ... }

    // True se tutti e 4 gli slot sono OCCUPIED
    public boolean isFull() { ... }

}
Il testo non specifica con quale criterio associare uno slot al container. Possibili strategie: primo libero (first free), oppure ottimizzazione del percorso del robot (minima distanza dallo IOPort).
Domanda: quale strategia di assegnazione degli slot deve essere adottata?
Risposta: basta prenderne uno libero (strategia first free).

HOME

HOME è la posizione iniziale del cargorobot nella stiva. Il robot deve tornare in HOME al termine di ogni operazione di carico. HOME non può essere un punto matematico ma deve essere una zona, correlata alla dimensione fisica del robot: la sua definizione e la logica di navigazione verso di essa sono parte del progetto del cargorobot che dobbiamo realizzare.

Stati del sistema cargoservice

Dall'analisi dei requisiti emergono i seguenti stati operativi del servizio:

StatoDescrizione
IdleIn attesa di richieste
EngagedRichiesta accettata, in attesa che il cliente porti il container (timeout 30 s) LED → lampeggia
DisengagedTimeout di 30 s scaduto
Service workingRobot in movimento, operazione di carico in corso
Out of serviceGuasto sonar rilevato (D > DFREE per ≥ 3 s)
Domanda: Cosa avviene dopo lo scadere del timer in stato Disengaged?
Risposta: si processa una nuova richiesta, "dimenticando" quella precedente: il sistema torna semplicemente disponibile in idle.
Domanda: Durante lo stato Service working, il LED è spento oppure lampeggia?
Risposta: il sistema è ancora da considerarsi engaged, quindi il LED continua a lampeggiare.

Struttura del sistema

Il cargoservice è un sistema eterogeneo: ogni nodo computazionale impiega un tipo di software diverso (applicazione, firmware su microcontrollore, simulatore del robot). Java da solo non basta a coprire lo scambio di messaggi; per questo si adotta qak, un linguaggio di modellazione, pensato proprio per distinguere i tipi di interazione fra le entità (dispatch, request, event). Il modello qak completo sarà sviluppato nella Problem Analysis, ma i nodi principali sono già individuabili a livello di requisiti:

  • Nodo principale: cargoservice
  • Nodo robot: cargorobot, che si appoggia al servizio VirtualRobot26
  • Microcontrollore sonar: RaspBerry PicoW con firmware da sviluppare
  • Dispositivo IOPort: nodo separato che espone la web GUI (pushbutton + display) al cliente

Si contano quindi almeno 3 nodi (virtualrobot, cargoservice, sonar), 4 se si considera l'IOPort come nodo a sé stante.

Natura delle entità: attori vs. POJO

Il punto di partenza è decidere la natura computazionale di ciascuna entità identificata nell'analisi dei requisiti. Un POJO (Plain Old Java Object) è un oggetto passivo; un attore è un'entità autonoma, reattiva su messaggi in arrivo e può inviarne ad altri.

Il cargoservice deve reagire a eventi concorrenti (una pressione sul pushbutton, una lettura del sonar, un segnale del marker, una risposta del robot) che possono arrivare in qualsiasi momento e in qualsiasi ordine. Questa natura reattiva e concorrente rende il modello ad attori la scelta più adatta per il sistema: ciascuna entità gestisce la propria coda di messaggi e reagisce agli stimoli senza bloccare le altre.

EntitàNaturaMotivazione
cargoserviceAttore qakOrchestratore principale: reattivo a loadrequest, sonardata, markdone, robotdone
cargorobotAttore qakEntità autonoma con comportamento sequenziale proprio; riceve comandi e risponde al termine
sonarAttore qakRiceve continuamente stringhe grezze dal PicoW e le trasforma in eventi comprensibili al sistema
ioportAttore qakGestisce gli eventi del pushbutton e aggiorna il display; disaccoppia l'hardware dall'orchestratore
HoldPOJOStruttura dati passiva (stato degli slot) gestita internamente dal cargoservice

Gestione del sonar

L'uso di emit (broadcast) anziché dispatch è motivato dal fatto che il broadcast disaccoppia il produttore dai consumatori rendendolo technology independent: il sonar emette un evento senza assumere nulla su chi lo riceverà né su come lo elaborerà, permettendo di aggiungere nuovi consumatori senza toccare il produttore.

Modello qak (architettura logica)

Il committente mette a disposizione più varianti di VirtualRobot26: RobotSmart26, RobotService26, RobotObj26. Si sceglie la variante che riduce maggiormente l'abstraction gap, ovvero RobotSmart26: è in grado di calcolare autonomamente il percorso verso un punto, ricevendo in input solo le coordinate di destinazione. Il cargorobot lo utilizza come servizio esterno.

Anche RobotSmart26 espone comunque solo primitive di basso livello rispetto ai requisiti che richiedono invece operazioni di alto livello: raggiungere l'IOPort, prelevare il container, raggiungere slot5, etc.

Questo abstraction gap viene colmato definendo un livello intermedio di navigazione, che traduce le operazioni di alto livello in sequenze di comandi primitivi, mantenendo la responsabilità di cosa fare nel cargoservice e quella di come muoversi nel cargorobot.

Il modello qak del sistema è il seguente. Distinguiamo chiaramente i tipi di messaggio:


request  loadrequest  : loadrequest(none)       //fa una richiesta di carico
reply    loadaccepted : loadaccepted(SLOTNAME)  //carico accettato 
reply    loadrejected : loadrejected(REASON)   //carico rifiutato (REASON: full | retrylater)

dispatch moveto       : moveto(DEST)        //ordina al robot di muoversi a DEST
dispatch pick         : pick(none)          //ordina al robot di prelevare il container
dispatch place        : place(none)         //ordina di rilasciare il container
dispatch halt         : halt(none)          //ordina al robot di fermarsi
dispatch resume       : resume(none)        //ordina al robot di ripartire
dispatch robotdone    : robotdone(none)     //robot termina operazione
dispatch markdone     : markdone(none)      //marker terminato
dispatch updateDisplay: updateDisplay(MSG)  //aggiorna il display con un messaggio MSG
dispatch blinkLed     : blinkLed(STATE)     //fa lampeggiare il led (STATE: ON | OFF)

event sonarcontainer  : sonarcontainer(D)   //notifica che c'è un container disponibile per il ritiro
event sonarfault      : sonarfault(D)       //notifica che c'è un problema
event sonarfree       : sonarfree(D)        //notifica che non ci sono container in attesa sulla porta

context ctxcargoservice ip [host="localhost" port=8050]

}

Test Plans

L'unico componente testabile in isolamento è la classe Hold. I test verificano la logica richiesta dai requisiti: accettazione/rifiuto richiesta e stato della stiva.

T1 — Hold

T1.1 A partire da una hold vuota, reserveFirstFree() deve restituire "slot1", applicando la strategia first free.

T1.2 Dopo aver riservato e marcato come occupati tutti e 4 gli slot, isFull() deve restituire true.

T1.3 Con la hold piena, reserveFirstFree() deve restituire null, indicando che non è possibile accettare ulteriori richieste (R1: rejects the request when the hold is full).

T1.4 Dopo una reserveFirstFree() seguita da markReservedOccupied(), lo slot deve risultare OCCUPIED e la hold non ancora piena; una successiva reserveFirstFree() deve riuscire (restituisce non-null).

T1.5 Con uno slot già RESERVED, una seconda chiamata a reserveFirstFree() deve restituire null: la hold ammette al più una prenotazione attiva alla volta.

class HoldTest {

    Hold hold;

    @BeforeEach
    void setUp() { hold = new Hold(); }

    @Test // T1.1
    void reserveFromEmpty() {
        assertEquals("slot1", hold.reserveFirstFree());
    }

    @Test // T1.2
    void fullWhenAllOccupied() {
        for (int i = 0; i < Hold.NUM_SLOTS; i++) {
            hold.reserveFirstFree();
            hold.markReservedOccupied();
        }
        assertTrue(hold.isFull());
    }

    @Test // T1.3
    void reserveWhenFull() {
        for (int i = 0; i < Hold.NUM_SLOTS; i++) {
            hold.reserveFirstFree();
            hold.markReservedOccupied();
        }
        assertNull(hold.reserveFirstFree());
    }

    @Test // T1.4
    void markReservedBecomesOccupied() {
        hold.reserveFirstFree();
        hold.markReservedOccupied();
        assertFalse(hold.isFull()); // 1/4, non piena
        assertNotNull(hold.reserveFirstFree());
    }

    @Test // T1.5
    void noDoubleReserve() {
        hold.reserveFirstFree(); // slot1 → RESERVED
        assertNull(hold.reserveFirstFree());
    }
}
