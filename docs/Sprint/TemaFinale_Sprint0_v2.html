Introduction

Goal: Costruire un sistema software cargoservice che automatizza le operazioni di carico di container nella stiva di una nave, impiegando un robot a guida differenziale (cargorobot).

Requirements

The company asks us to build a service named cargoservice that:

R1   is able to receive a request to load a container sent by some customer via the pushbutton of the IOPort.

  • It sends the answer retrylater if the IOPort is currently occupied by a container, or if the system is Out of service.
  • It rejects the request when the hold is already full (i.e. slots 1–4 are all occupied).
  • Otherwise, it considers the system as engaged, detects a free slot, and returns the name of the reserved slot. While engaged, the system must blink a LED.

R2   when the request to load is accepted, the customer must move the container into the sensor area within a prefixed time (e.g. 30 secs); otherwise the system becomes disengaged.

R3   uses the cargorobot to move the container from the IOPort to slot5 (for barcode marking the container) and then to the reserved slot.

R4   shows on the display of the IOPort:

  • the current state of the hold;
  • the message "Service working" when everything is going well;
  • the message "Out of service" if the sonar measures D > DFREE for at least 3 secs (possible sonar failure).

Requirement Analysis

Sistema cargoservice

Diamo il nome cargoservice al sistema software da realizzare, per distinguerlo dalle entità che lo compongono. Si tratta di un sistema distribuito su più nodi computazionali, i cui componenti operano tramite scambio di messaggi.

Il sistema comprende le seguenti macro-entità:

  • il cargorobot (DDR), da progettare e realizzare
  • il sensore sonar, dispositivo fisico da integrare
  • la stiva (hold), con i suoi slot di posizionamento
  • il marker, dispositivo che etichetta con barcode il container in slot5 e segnala il completamento
  • il dispositivo IOPort, dotato di pushbutton e display
  • il LED di allarme/stato, attuatore fisico

cargorobot (DDR)

Il DDR è un robot a guida differenziale che può muoversi in avanti, indietro, ruotare su se stesso e fermarsi. È disponibile un simulatore (VirtualRobot26) che espone i comandi-base del robot in cril, presentandosi come un servizio che fornisce solo primitive di movimento.

Il cargorobot come entità con comportamento applicativo va però progettato e realizzato da noi: la responsabilità dell'orchestrazione delle operazioni (raggiungere l'IOPort, prelevare il container, raggiungere slot5, ecc.) appartiene al cargoservice, non al DDR. Esiste dunque un abstraction gap significativo tra i comandi primitivi del simulatore e le operazioni di alto livello richieste, che sarà analizzato nella Problem Analysis.

Domanda: Il servizio VirtualRobot26 è sufficiente per risolvere il problema (posizionare un container in uno slot e tornare in HOME) o richiede modifiche?
Risposta: si assume che il simulatore fornisca le primitive necessarie (avanza, ruota, fermati) e che la logica applicativa vada implementata sopra di esse.

IOPort

L'IOPort è un dispositivo fisico collocato all'ingresso della stiva. È composto da:

  • un pushbutton: premuto dal cliente per inviare una richiesta di carico;
  • un display: mostra la risposta alla richiesta e lo stato corrente della stiva;

Sonar sensor

Il sonar è un sensore fisico collocato davanti all'IOPort. Misura continuamente una distanza D. Il valore DFREE è una costante di configurazione che rappresenta la distanza libera attesa (area vuota davanti al sensore).

Due soglie operative:

  • D < DFREE/2 per almeno 3 secondi → presenza di un container allo IOPort → stato Service working
  • D > DFREE per almeno 3 secondi → possibile guasto del sensore → stato Out of service
Domanda: Il software che gestisce il sonar è già disponibile, oppure dobbiamo costruirlo noi?
Risposta: in attesa di uno strumento più evoluto, viene fornito un dispositivo fisico basato su un RaspBerry PicoW, senza software di supporto. Occorre scrivere il firmware sul microcontrollore.
Domanda: in quale formato ci aspettiamo di ricevere i dati dal sonar?
Risposta: stringhe della forma distance(DIRECTION, D), dove D è la distanza misurata (intero, in cm o mm).

Occorre valutare se sia sufficiente scrivere software sul microcontrollore per integrare compiutamente il sonar nel sistema, oppure se serva un componente intermediario (proxy/adapter) che traduca i messaggi grezzi in eventi comprensibili al cargoservice.

Domanda: Cosa accade se il guasto sonar si verifica mentre il cargorobot è già in movimento? Due interpretazioni: (A) il robot si ferma immediatamente e riprende dal punto in cui era al ripristino; (B) il robot completa l'operazione corrente e poi il sistema si mette in attesa.
Risposta: interpretazione A. Il robot si ferma immediatamente (halt), il container rimane nella posizione corrente e il sistema resta sospeso finché il sonar non torna a misurare D ≤ DFREE. Al ripristino, il robot riprende l'operazione dal punto in cui si era fermato.

Marker (slot5)

slot5 è una stazione di marcatura temporanea. Quando il cargorobot deposita un container in slot5, il dispositivo marker applica un barcode identificativo e segnala al sistema quando l'attività è completata. Solo allora il robot può prelevare il container e portarlo nello slot riservato.

Domanda: Come segnala il marker il completamento della marcatura? Via messaggio su rete?

Hold (stiva)

La stiva è un'area rettangolare e piatta con un unico punto di ingresso/uscita (IOPort). Fornisce:

  • slot1–slot4: slot per la deposizione definitiva dei container
  • slot5: stazione temporanea di marcatura

Formalizziamo la stiva come classe Java Hold:

public class Hold {
    public static final int NUM_SLOTS = 4;       // slot1..slot4

    public enum SlotState { FREE, RESERVED, OCCUPIED }
    // RESERVED: slot assegnato a una richiesta accettata, robot non ancora arrivato.
    // Finché uno slot è RESERVED il sistema è 'engaged' e non accetta nuove richieste.

    private SlotState[] slots = { // slot1..slot4
        SlotState.FREE,
        SlotState.FREE, 
        SlotState.FREE,
        SlotState.FREE
    };

    // slot5 è gestito separatamente: è libero o occupato dal marker
    private boolean slot5Occupied = false;
}
Il testo non specifica con quale criterio associare uno slot al container. Possibili strategie: primo libero (first free), oppure ottimizzazione del percorso del robot (minima distanza dallo IOPort).
Domanda: quale strategia di assegnazione degli slot deve essere adottata?
Risposta: basta prenderne uno libero (strategia first free).

HOME

HOME è la posizione iniziale del cargorobot nella stiva. Il robot deve tornare in HOME al termine di ogni operazione di carico. HOME non può essere un punto matematico ma deve essere una zona, correlata alla dimensione fisica del robot: la sua definizione e la logica di navigazione verso di essa sono parte del progetto del cargorobot che dobbiamo realizzare.

Struttura del sistema

Il cargoservice è un sistema distribuito sui seguenti nodi computazionali:

  • Nodo principale: cargoservice (orchestrazione, business logic)
  • Nodo robot: cargorobot (DDR + VirtualRobot26)
  • Microcontrollore sonar: RaspBerry PicoW con firmware da sviluppare
  • Dispositivo IOPort: pushbutton + display + sonar integrati
  • Marker: dispositivo di marcatura in slot5

La struttura del sistema sarà formalizzata tramite un modello qak, che cattura i requisiti distinguendo i tipi di interazione tra le entità (messaggi punto-a-punto, eventi broadcast, interazioni sincrone). Il modello qak sarà sviluppato nella Problem Analysis.

Problem Analysis

L'analisi del problema ha l'obiettivo di passare da cosa il sistema deve fare (requisiti) a come è strutturato: quali entità computazionali lo compongono, che natura hanno, e come si scambiano messaggi. Il risultato è un modello qak che cattura i requisiti in termini di attori.

Natura delle entità: attori vs. POJO

Il punto di partenza è decidere la natura computazionale di ciascuna entità identificata nell'analisi dei requisiti. Un POJO (Plain Old Java Object) è un oggetto passivo; un attore è un'entità autonoma, reattiva su messaggi in arrivo e può inviarne ad altri.

Il cargoservice deve reagire a eventi concorrenti (una pressione sul pushbutton, una lettura del sonar, un segnale del marker, una risposta del robot) che possono arrivare in qualsiasi momento e in qualsiasi ordine. Questa natura reattiva e concorrente rende il modello ad attori la scelta più adatta per il sistema: ciascuna entità gestisce la propria coda di messaggi e reagisce agli stimoli senza bloccare le altre.

EntitàNaturaMotivazione
cargoserviceAttore qakOrchestratore principale: reattivo a loadrequest, sonardata, markdone, robotdone
cargorobotAttore qakEntità autonoma con comportamento sequenziale proprio; riceve comandi e risponde al termine
sonarproxyAttore qakRiceve continuamente stringhe grezze dal PicoW e le trasforma in eventi comprensibili al sistema
ioporthandlerAttore qakGestisce gli eventi del pushbutton e aggiorna il display; disaccoppia l'hardware dall'orchestratore
HoldPOJOStruttura dati passiva: stato degli slot, peso corrente; gestita internamente dal cargoservice
Domanda: Il dispositivo marker, va inteso come dispositivo esterno (hardware) che segnala il comportamento via messaggio o come un servizio?

Abstraction gap del cargorobot

Il simulatore VirtualRobot26 espone solo primitive di basso livello: muovi avanti per N passi, ruota a sinistra/destra, fermati. I requisiti richiedono invece operazioni di alto livello: raggiungere l'IOPort, prelevare il container, raggiungere slot5, depositare, raggiungere lo slot riservato, tornare a HOME.

Questo abstraction gap viene colmato definendo un livello intermedio di navigazione, che traduce le operazioni di alto livello in sequenze di comandi primitivi, mantenendo la responsabilità di cosa fare nel cargoservice e quella di come muoversi nel cargorobot.

Il cargorobot espone quindi un'interfaccia a messaggi, del tipo:

// Messaggi che cargoservice invia a cargorobot (dispatch)
dispatch moveto      : moveto(DESTINATION)   // es. moveto(IOPORT), moveto(SLOT1), moveto(HOME)
dispatch pick        : pick(none)
dispatch place       : place(none)
dispatch halt        : halt(none)
dispatch resume      : resume(none)

// Messaggi che cargorobot risponde a cargoservice (dispatch)
dispatch robotdone   : robotdone(OK)
dispatch robotfailed : robotfailed(REASON)

Il fatto che moveto riceva una destinazione simbolica (e non coordinate fisiche) è una scelta deliberata di incapsulamento: la mappatura da nome-slot a percorso fisico appartiene al cargorobot, che conosce la geometria della stiva.

Gestione del sonar: proxy e temporizzazione

Il sonar sul PicoW emette continuamente stringhe distance(DIRECTION, D). Il cargoservice non può consumare direttamente questi dati grezzi: occorre un attore sonarproxy che:

  • riceve le stringhe via UDP/TCP dal microcontrollore;
  • applica la logica di temporizzazione (3 secondi sotto soglia / sopra soglia);
  • emette eventi broadcast quando le condizioni sono soddisfatte.

L'uso di emit (broadcast) anziché dispatch è motivato dal fatto che la lettura sonar riguarda potenzialmente più attori: il cargoservice (per rilevare il container) e lo stesso cargoservice per il fault, ma in futuro potrebbe riguardare anche un logger o un monitor. Il broadcast disaccoppia il produttore dai consumatori rendendolo technology independent: il sonarproxy emette un evento senza assumere nulla su chi lo riceverà né su come lo elaborerà, permettendo di aggiungere nuovi consumatori senza toccare il produttore.

// Eventi emessi da sonarproxy (broadcast)
event sonarcontainer : sonarcontainer(D)   // D < DFREE/2 per >= 3 s: container presente
event sonarfault     : sonarfault(D)       // D > DFREE per >= 3 s: possibile guasto
event sonarfree      : sonarfree(D)        // ripristino dopo fault

Flusso principale e gestione degli stati

Il cargoservice è un attore con stato esplicito. I suoi stati operativi corrispondono direttamente ai requisiti e guidano la struttura del modello qak.

// Stati interni del cargoservice
State idle           // in attesa di richieste
State engaged        // richiesta accettata, attende container (timeout 30s), LED blink
State disengaged     // timeout scaduto senza che il container sia stato rilevato
State serviceworking // robot in movimento, operazione di carico in corso
State outofservice   // fault sonar: tutte le richieste rifiutate con 'retrylater'

In stato idle il sistema è disponibile. All'arrivo di una loadrequest il cargoservice verifica la hold: se piena risponde loadrejected(full), altrimenti riserva il primo slot libero, risponde con il suo nome, avvia il LED in modalità blink e fa partire il timer da 30 secondi. Il sistema entra in stato engaged.

// [idle] → ricezione richiesta di carico
Transition t0 whenMsg loadrequest -> engaged

State engaged {
    [# val slot = hold.reserveFirstFree()
       replyTo(loadrequest, loadaccepted(slot))
       ledBlink() #]
    delay 30000   // timeout: il cliente ha 30s per portare il container
}

In stato engaged il sistema attende che il sonar rilevi il container entro il timeout. Se arriva l'evento sonarcontainer in tempo, il cargoservice ordina al robot di raggiungere l'IOPort e transisce in serviceworking. Se invece scade il timer senza rilevamento, lo slot viene rilasciato e il sistema torna in idle (disengaged è uno stato transitorio di pulizia).

// [engaged] → container rilevato in tempo
Transition t1
    whenEvent sonarcontainer -> serviceworking
    whenTime  30000          -> disengaged

State serviceworking {
    forward cargorobot -m moveto(IOPORT)
}

State disengaged {
    [# hold.releaseReserved(); ledOff() #]
}
Goto idle

In stato serviceworking il cargoservice orchestra la sequenza di operazioni del robot. Ogni passo è atomico: il sistema invia un comando e attende il corrispondente robotdone prima di procedere. Il robot raggiunge l'IOPort e preleva il container.

// [serviceworking] — fase 1: prelievo all'IOPort
Transition t2 whenMsg robotdone -> pickAtIoport

State pickAtIoport {
    forward cargorobot -m pick(none)
}
Transition t3 whenMsg robotdone -> moveToSlot5

State moveToSlot5 {
    forward cargorobot -m moveto(SLOT5)
}
Transition t4 whenMsg robotdone -> placeAtSlot5

Il container viene depositato in slot5 per la marcatura. A differenza degli altri passi, qui il sistema non attende robotdone ma markdone: è il marker a segnalare il completamento dell'etichettatura, non il robot. Solo dopo quel segnale il robot può riprendere.

// [serviceworking] — fase 2: marcatura in slot5
State placeAtSlot5 {
    forward cargorobot -m place(none)
}
Transition t5 whenMsg markdone -> pickFromSlot5   // attende il marker, non robotdone

State pickFromSlot5 {
    forward cargorobot -m pick(none)
}
Transition t6 whenMsg robotdone -> moveToReserved

Il container marcato viene portato nello slot riservato e depositato. Il robot infine torna in HOME: solo al ricevimento dell'ultimo robotdone la hold viene aggiornata, il LED spento e il sistema torna disponibile in idle.

// [serviceworking] — fase 3: deposizione nello slot riservato e ritorno
State moveToReserved {
    [# val dest = hold.getReservedSlot() #]
    forward cargorobot -m moveto(dest)
}
Transition t7 whenMsg robotdone -> placeAtReserved

State placeAtReserved {
    forward cargorobot -m place(none)
}
Transition t8 whenMsg robotdone -> returnHome

State returnHome {
    forward cargorobot -m moveto(HOME)
}
Transition t9 whenMsg robotdone -> idle   // hold aggiornata, LED off

Gestione del fault sonar (interruzione e ripresa)

L'evento sonarfault può arrivare in qualsiasi stato. Il cargoservice lo gestisce come transizione trasversale:

  • se in working: invia halt al robot, salva lo stato corrente dell'operazione (quale passo era in corso), accende LED fisso, mostra "Out of service" sul display;
  • all'arrivo di sonarfree: invia resume al robot (che riprende dal punto in cui era), torna allo stato working;
  • se in idle o engaged: risponde retrylater a eventuali richieste, attende sonarfree.

Modello qak (architettura logica)

Il modello qak del sistema è il seguente. Distinguiamo chiaramente i tipi di messaggio:

// ── MESSAGGI ──────────────────────────────────────────────────
// Richieste sincrone (request/reply)
request  loadrequest  : loadrequest(none)
reply    loadaccepted : loadaccepted(SLOTNAME)
reply    loadrejected : loadrejected(REASON)   // REASON: full | retrylater

// Comandi punto-a-punto (dispatch, fire-and-forget)
dispatch moveto       : moveto(DEST)
dispatch pick         : pick(none)
dispatch place        : place(none)
dispatch halt         : halt(none)
dispatch resume       : resume(none)
dispatch robotdone    : robotdone(none)
dispatch markdone     : markdone(none)
dispatch updateDisplay: updateDisplay(MSG)
dispatch blinkLed     : blinkLed(ONOFF)

// Eventi broadcast (emit)
event sonarcontainer  : sonarcontainer(D)
event sonarfault      : sonarfault(D)
event sonarfree       : sonarfree(D)

// ── ATTORI ────────────────────────────────────────────────────
context ctxCargoservice ip [host="localhost" port=8050]

QActor cargoservice context ctxCargoservice {
    // gestisce loadrequest, sonarcontainer, sonarfault, sonarfree,
    // robotdone, markdone; orchestra il sistema
}

QActor cargorobot context ctxCargoservice {
    // riceve moveto/pick/place/halt/resume
    // invia robotdone al termine di ogni operazione
}

QActor sonarproxy context ctxCargoservice {
    // legge dati grezzi dal PicoW, emette sonarcontainer / sonarfault / sonarfree
}

QActor ioporthandler context ctxCargoservice {
    // legge eventi pushbutton, invia loadrequest a cargoservice
    // riceve updateDisplay, aggiorna il display fisico
}

La separazione tra cargoservice (orchestratore) e ioporthandler / sonarproxy rispetta il principio di separazione delle responsabilità (SOC): il servizio applicativo non dipende dai dettagli del protocollo hardware.

Test Plans

I test sono organizzati per requisito e per stato del sistema. L'obiettivo è verificare il comportamento del cargoservice in modo incrementale: prima il nucleo logico (gestione della hold, accettazione/rifiuto richieste), poi l'integrazione con il robot, poi i casi eccezionali. I test sul comportamento temporale (timeout, soglie sonar) usano costanti configurabili per permettere esecuzioni rapide in ambiente di test.

Strategia di esecuzione

I test T1 non richiedono hardware né simulatore robot: la hold è un POJO testabile in isolamento con JUnit, stimolando direttamente l'attore cargoservice via messaggi in-memory.

I test T2 richiedono il sonarproxy con un mock del PicoW (un thread che invia stringhe distance a frequenza configurabile).

I test T3–T5 richiedono VirtualRobot26 attivo. Per T5.3 e T5.4 occorre iniettare il fault sonar in modo controllato durante l'esecuzione del robot: si usa il mock del PicoW con timing preciso.

Tutte le costanti temporali (timeout 30 s, soglia 3 s sonar) sono esternalizzate in un file di configurazione e abbassabili a valori ridotti (es. 1 s) per i test automatici.
